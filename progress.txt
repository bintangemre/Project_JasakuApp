## 2026-06-29 — P1-7 FIX TWO 500 ERRORS ON ORDER SUBMIT

### Fixed (Round 1: PostGIS + Notification crash)
- jasaku-backend/src/modules/orders/orders.service.ts: Ganti `tx.order_locations.create({ location: { type: "Point", ... } })` dengan raw SQL `ST_SetSRID(ST_MakePoint(...))` karena field `location` adalah `Unsupported("geometry")` di Prisma — Prisma tidak bisa handle PostGIS geometry via `.create()`
- jasaku-backend/src/modules/orders/orders.service.ts: Bungkus `NotificationService.sendToUser` di try-catch agar kegagalan notifikasi (misal Firebase belum dikonfigurasi) tidak menggagalkan pembuatan order

### Fixed (Round 2: FK constraint — wrong ID type)
Root cause: Schema `orders.customer_id → profiles_customer.id` dan `orders.provider_id → provider_profiles.id`, tapi semua kode pakai `users.id` (dari JWT) untuk field2 tsb. Karena setiap user punya profile dengan UUID berbeda, FK constraint selalu violated.

- **createOrder**: lookup `profiles_customer.id` & `provider_profiles.id` dari `users.id` sebelum transaksi, pakai profile IDs di `orders.create`
- **getCustomerOrders**: translate `users.id` → `profiles_customer.id` sebelum query
- **getProviderOrders**: translate `users.id` → `provider_profiles.id` sebelum query
- **receiveOrderStatus**: translate `users.id` → `provider_profiles.id` untuk comparison & active order check; juga fix notifikasi pakai `users.id` (bukan profile ID)
- **getTodayOrders**: translate `users.id` → `provider_profiles.id` sebelum query
- **cancelOrder**: translate `users.id` → `profiles_customer.id` untuk comparison; fix notifikasi pakai `users.id`
- Semua notifikasi `sendToUser` sekarang dikirim dengan `users.id` karena `user_devices.user_id → users.id`
